**2. Описание назначения**

2.1. Цель -- усовершенствовать и оптимизировать процессы банка, а также повысить эффективность работы пользователей путём внедрения технологий, основанных на искусственном интеллекте (далее --- ИИ).

2.2. Программное обеспечение, предназначенное для внедрения и использования в банковских процессах инструментов (сервисов, модулей, функциональностей), основанных на технологии искусственного интеллекта (далее -- ПО) должно включать в себя:

2.2.1. модуль на основе генеративных нейронных сетей, который может интегрироваться на уровне нижеперечисленных модулей (модуль LLM);

2.2.2. модуль Контакт-центра, включающий в себя:

2.2.2.1. модуль понимания запросов;

2.2.2.2. интерфейс для работы суфлёра для помощи операторам канала телефония;

2.2.2.3. интерфейс для работы суфлёра для помощи операторам онлайн-чата;

2.2.2.4. модуль онлайн-чат для работы операторов онлайн-чата;

2.2.3. интерфейс для работы модуля ИИ-ассистента для помощи пользователям;

2.2.4. модуль распознавания документов.

2.3. ПО должно выполнять следующие функции:

2.3.1. для модуля Контакт-центра:

2.3.1.1. поддержка операторов на канале телефония в режиме суфлёра, который обеспечивает помощь операторам канала телефония в режиме реального времени для обработки входящих и исходящих звонков, повышение скорости и точности ответов;

2.3.1.2. автоматическая выдача релевантных ответов модуля LLM операторам онлайн-чата в режиме суфлёра в ходе ведения диалога, чтобы ускорить взаимодействие с клиентами;

2.3.1.3. доступ внутренних пользователей Контакт-центра в модуле Контакт-центра к диалоговому окну с ИИ для проверки корректности и релевантности предоставляемых ответов языковой моделью, для оценки и доработки ИИ в контролируемой среде;

2.3.1.4. предоставление ответов модулем LLM на русском и на английском языках исходя из соответствующего языка запроса клиента;

2.3.2. для модуля ИИ-ассистент:

2.3.2.1. предоставление релевантных ответов модуля LLM пользователям в рамках использования ИИ-ассистента;

2.3.3. для модуля распознавания документов:

2.3.3.1. автоматическое распознавание, классификацию и извлечение структурированной информации из различных типов документов, поступающих в банк в виде фотографий, PDF-файлов;

2.3.3.2. использование технологий компьютерного зрения и искусственного интеллекта для повышения точности и скорости обработки документов.

## 3. Общие требования к модулю LLM

3.1. ПО должно поддерживать длину загружаемого контекста генеративной LLM не менее 8 200 токенов/символов.

3.2. ПО должно поддерживать возможность подключения одной или нескольких LLM для каждого отдельного модуля ПО для неограниченного количества баз знаний LLM, внутренних или внешних источников данных.

3.3. ПО должно поддерживать возможность установки следующих параметров модели:

3.3.1. температура (параметр, регулирующий случайность сгенерированного результата - высокая температура способствует более разнообразным и творческим ответам, низкая температура - более консервативным и детерминированным результатам);

3.3.2. максимальная длина ответа в токенах/символах;

3.3.3. длина фрагмента в символах, на которые делится документ;

3.3.4. длина перекрытия фрагментов между собой;

3.3.5. семантическая близость для включения фрагмента в контекст для предоставления ответа;

3.3.6. семантическая близость, при которой ответ предоставляется с помощью детерминированных (готовых) ответов из базы знаний LLM без генерации.

3.4. Модуль LLM должен предоставлять генеративные ответы длиной до 500 символов в течение 1-2 секунд или быстрее при одновременной нагрузке не менее 10 запросов в секунду.

3.5. Модуль LLM должен генерировать ответы с учетом дополнительно найденной релевантной информации с помощью ретривера.

3.6. Модуль LLM должен обеспечивать низкий уровень галлюцинаций -- не более 3%.

## 44.2. Требования к модулю понимания запросов модуля Контакт-центра

## 4.2.1. Общие требования к модулю понимания запросов 

4.2.1.1. Модуль Контакт-центра должен включать в себя модуль понимания запросов пользователей. Модуль понимания запросов должен предоставлять возможность его настройки (управления) работниками Заказчика без навыков использования языков программирования.

4.2.1.2. Модуль понимания запросов должен работать исключительно в инфраструктуре банка без доступа в сеть Интернет.

4.2.1.3. Для обучения модуль понимания запросов должен использовать обучающую выборку документов баз знаний LLM -- накопленные в каждом документе баз знаний LLM примеры вопросов пользователей, каждый новый вопрос пользователя должен автоматически сопоставляться с накопленным массивом примеров вопросов и ответов.

4.2.1.4. Для работы модуля понимания запросов (распознавания синонимичных слов и словосочетаний, полной смены формулировки на семантически близкую) должно быть достаточно одной формулировки вопроса.

4.2.1.5. Модуль понимания запросов должен поддерживать работу с русским и английским языками.

4.2.1.6. Модуль понимания должен автоматически учитывать порядок слов, использование синонимов или антонимов; результат каждого поиска в базе знаний LLM должен быть снабжен показателем релевантности в результатах поиска, отражающим вероятность того, что найденный документ базы знаний LLM соответствует вопросу пользователя.

4.2.1.7. Обучение модуля понимания не должно требовать указания ключевых слов или какой-либо другой ручной разметки примеров вопросов.

4.2.1.8. Обучение модуля понимания должно запускаться автоматически сразу после каждого обновления базы знаний LLM.

4.2.1.9. Добавление новых формулировок вопросов должно происходить полностью автоматически или через ручную валидацию администратором базы знаний LLM.

4.2.1.10. Список примеров в обучающей выборке должен также иметь возможность пополняться через загрузку СУЗ.

4.2.1.11. При добавлении новых формулировок модуль понимания должен автоматически контролировать наличие повторяющихся примеров в обучающей выборке и при пополнении обучающей выборки повторяющимся примером вопроса заблокировать его добавление.

4.2.1.12. Администратор базы знаний LLM должен иметь инструмент контроля над работой модуля понимания, который позволит определить какие документы будут найдены по запросу пользователя, с каким процентом релевантности и из какого примера вопроса.

4.2.1.13. Администратор базы знаний LLM должен иметь возможность указать минимальную релевантность поиска, начиная с которой модуль Контакт-центра может использовать найденный документ для предоставления ответа пользователю.

4.2.1.14. Должна быть возможность уточнения вопроса в случае, когда релевантности найденных документов недостаточно для прямого ответа, при этом должен предоставляться пользователю список вариантов ответа для выбора.

4.2.1.15. Администратор базы знаний LLM должен иметь возможность указать минимальную релевантность для вывода списка вариантов уточняющих вопросов клиенту.

4.2.1.16. Должна быть возможность включить проверку и настройку действий в случае одинаковых реплик пользователя, поступивших в диалоге несколько раз подряд.

4.2.1.17. Разработчиком должны быть обеспечены регулярные обновления и улучшения модуля понимания в течение гарантийного периода и технической поддержки.

4.2.2.11. Должна быть предоставлена возможность выгрузки нераспознанных записей диалогов для последующего анализа в модуль «Отчетность».

## 44.3. Требования к модулю суфлёр модуля Контакт-центра

## 4.3.1. Функции модуля суфлёр для канала телефония

4.3.1.1. В канале телефония модуль суфлёр должен включать функции модуля понимания запросов в канале телефония и модуля LLM.

4.3.1.2. Преобразование речи клиента в текст должно иметь возможность начинаться с момента соединения клиента с оператором канала телефония.

4.3.1.3. Должна обрабатываться речь клиента и оператора канала телефония в режиме реального времени (как во время входящих, так и исходящих телефонных вызовов), обеспечивая точный перевод речи в текст с привязкой к контексту разговора.

4.3.1.4. Должны мгновенно передаваться после преобразования речи в текст текстовые данные в модуль LLM для поиска информации в базе знаний LLM, формирования ответа и передачи его оператору канала телефония, тратя на данные действия в совокупности не более 1-2 секунд на каждую реплику.

4.3.1.5. Модуль Контакт-центра должен извлекать информацию из базы знаний LLM и возвращать ответ, подходящий к запросу.

4.3.1.6. Передача ответа оператору канала телефония должна происходить в окне разговора с клиентом, которое видит оператор канала телефония с момента принятия вызова либо осуществления исходящего вызова.

4.3.1.7. Помимо основного ответа ПО должно предоставлять дополнительные материалы исходя из контекста запроса: шаблоны, документы, таблицы, изображения, чтобы оператор канала телефония мог адаптировать ответ под ситуацию клиента.

4.3.1.8. К ответу дополнительно должна предоставляться ссылка на используемый раздел, статью СУЗ или несколько ссылок.

4.3.1.9. Должна фиксироваться обратная связь по предоставленному ответу модуля LLM путём нажатия оператором канала телефония в окне разговора с клиентом одной из кнопок: «воспользовался», «не воспользовался», «неполный ответ» или иные.

4.3.1.10. При входящем и исходящем звонке должен определяться язык клиента для оператора канала телефония (русский или английский) с высокой точностью -- не менее 99% с возможностью переключения с одного языка на другой в режиме реального времени.

4.3.1.11. Ответ должен генерироваться на том же языке, что и входящий запрос. В случае, если запрос клиента на английском языке, оператору канала телефония должен предоставляться:

4.3.1.11.1. текст вопроса клиента и его перевод;

4.3.1.11.2. ответ на английском языке: для непосредственного зачитывания клиенту;

4.3.1.11.3. ответ на русском языке: для проверки корректности ответа и понимания оператором канала телефония информации.

4.3.1.12. В случае, если запрос клиента не распознан, должно выводиться сообщение оператору канала телефония - «запрос не распознан».

4.3.1.13. Модуль Контакт-центра должен подключаться к единой истории взаимодействий с клиентом во всех каналах коммуникации банка, анализировать её и при запросе оператора канала телефония отображать краткую выжимку (саммаризация) о предыдущих взаимодействиях с клиентом.

4.3.1.14. Ответ модуля LLM должен предоставляться клиенту на основе контекста предыдущих диалогов с данным клиентом с возможностью настройки автоматического использования или ручного вызова данного функционала.

## 4.3.2. Функции модуля суфлёр для канала онлайн-чат

4.3.2.1. В канале онлайн-чат модуль суфлёр должен включать функции модуля понимания запросов и модуля LLM.

4.3.2.2. Модуль суфлёр в канале онлайн-чат обрабатывает входящие текстовые сообщения клиентов, чтобы предоставить оператору онлайн-чата релевантный ответ.

4.3.2.3. Время обработки запроса с момента его поступления до предоставления ответа оператору онлайн-чата должно составлять не более 2 секунд.

4.3.2.4. Рабочее место операторов онлайн-чата должно включать в себя функционал получения ответов модуля LLM на основе базы знаний LLM. Ответы должны изменяться в зависимости от изменения тематики переписки оператора онлайн-чата и клиента. Ответы должны предоставляться оператору онлайн-чата в ранжированном виде от наибольшей релевантности к наименьшей с указанием процента релевантности.

4.3.2.5. Должна фиксироваться обратная связь путём нажатия оператором онлайн-чата одной из кнопок: «воспользовался», «не воспользовался», «неполный ответ» или иные.

4.3.2.6. Должна быть возможность для оператора онлайн-чата отправить клиенту ответ сразу, если он релевантен запросу. В противном случае должна быть возможность внести корректировки в ответ или позволить оператору онлайн-чата продолжить диалог самостоятельно, в том числе без дальнейшего участия модуля Контакт-центра.

4.3.2.7. Ответы модуля LLM должны отправляться клиентам без указания ссылок на используемые статьи СУЗ.

4.3.2.8. Должно формироваться оповещение оператору онлайн-чата о том, что клиенту будет отправлен сценарий диалога, а после оповещения модуль Контакт-центра должен начать автоматическую переписку с пользователем по выбранному диалоговому сценарию, если выбранный оператором онлайн-чата ответ содержит диалоговый сценарий.

4.3.2.9. Ответ должен генерироваться на том же языке, что и входящий запрос. В случае, если запрос клиента на английском языке, оператору онлайн-чата должен предоставляться:

4.3.2.9.1. текст вопроса клиента и его перевод;

4.3.2.9.2. ответ на английском языке: для непосредственного использования в ответе клиенту;

4.3.2.9.3. ответ на русском языке: для проверки корректности ответа и понимания оператором онлайн-чата информации.

4.3.2.10. В случае, если запрос клиента не распознан, должно выводиться сообщение оператору онлайн-чата - «запрос не распознан».

4.3.2.11. Модуль Контакт-центра должен подключаться к единой истории взаимодействий с клиентом во всех каналах коммуникации банка, анализировать её и при запросе оператора онлайн-чата отображать краткую выжимку (саммаризация) о предыдущих взаимодействиях с клиентом за выбранный промежуток времени в рамках периода хранения 1 год.

4.3.2.12. Ответ модуля LLM должен предоставляться клиенту на основе контекста предыдущих диалогов с данным клиентом с возможностью настройки автоматического использования или ручного вызова данного функционала.

4.4.44.3. модуль онлайн-чат должен предоставлять возможность устанавливать ограничения на размер и тип предаваемых файлов. Передаваемые в сообщениях от клиентов файлы должны проверяться антивирусным программным обеспечением банка до их использования (открытия, чтения, пересылки) операторами онлайн-чата и должны быть доступны для использования (открытия, чтения, пересылки) только в случае отсутствия угроз.

4.4.44.4. модуль онлайн-чат должен предоставлять возможность ограничения входа в личный кабинет пользователей по IP;

4.4.44.5. модуль онлайн-чат должен предоставлять возможность блокировки оператором/ пользователем клиентов, которые занимаются СПАМом.

4.4.45. Модуль онлайн-чат должен осуществлять отслеживание и фильтрацию операций, совершаемых пользователями (логирование).

4.4.46. Модуль онлайн-чат должен иметь возможность настройки приветственных сценариев для коммуникаций в канале онлайн-чат, сценариев при отсутствии ответа на запрос, настройки детерминированных ответов, стартового меню и т.п., если это допустимо с учетом технических возможностей канала.

4.4.47. Модуль онлайн-чат должен включать в себя возможность настройки работы очередей и функционала распределения диалогов.

4.4.48. Модуль онлайн-чат должен содержать роль (право доступа) позволяющее исключать и подключать операторов к работе с диалогами (направление диалогов оператору из очереди) в ручном режиме.

4.4.49. Модуль онлайн-чат при поступлении диалога в очередь операторам должен автоматически определять тематику диалога используя в качестве вариантов тематик названия статей СУЗ.

4.4.50. Модуль онлайн-чат должен предоставить администратору возможность выбора режима назначения диалогов на операторов -- общий список входящих диалогов для всех операторов или индивидуальный список каждого оператора.

4.4.51. Модуль онлайн-чат должен позволять администратору включить автоматическое назначение диалогов, при котором модуль должен помещать диалог в список входящих одного автоматически выбранного оператора.

4.4.52. Модуль онлайн-чат должен позволять администратору выбрать один из двух алгоритмов выбора оператора -- оператора с минимальной нагрузкой или оператора, который не получал диалог дольше других операторов.

4.4.53. Модуль онлайн-чат должен предоставлять возможность настройки максимального количества диалогов для каждого оператора при автоматическом распределении запросов по операторам в случае вхождения оператора в несколько групп одновременно, а именно, по среднему значению максимального количества диалогов из всех групп, в которые входит оператор, или по сумме максимального количества диалогов из всех групп, в которые входит оператор.

4.4.54. Модуль онлайн-чат должен отслеживать статус каждого оператора и не назначать диалоги на операторов, не находящихся в статусе «Online».

4.4.55. Модуль онлайн-чат должен предоставлять операторам возможность вернуть диалог в очередь, после чего модуль должен заново провести процедуру назначения диалога на оператора.

4.4.56. Модуль онлайн-чат должен при возврате диалога в очередь предоставить возможность выбрать другую группу операторов или сменить тематику диалога.

4.4.57. Модуль онлайн-чат должен предоставлять операторам возможность передать диалог напрямую другому оператору.

4.4.58. Разработчик должен предоставить свое программное решение для реализации модуля онлайн-чата для обслуживания клиентов или привлечь своего подрядчика для предоставления такого программного решения.

## 4.5. База знаний LLM для модуля Контакт-центра: разметка, работа со сценариями и промптами

## 4.5.1. База знаний LLM

4.5.1.1. Должна быть возможность добавлять и изменять промпты, сценарии, структуру и содержание базы знаний LLM без привлечения разработчика, без технических знаний, навыков, без знания языков программирования.

4.5.1.2. Должна быть возможность редактировать и структурировать данные базы знаний LLM так, чтобы их эффективно могла использовать LLM-модель.

4.5.1.3. Должна обновляться база знаний LLM в соответствии с изменениями СУЗ любыми способами, например, API, учитывая, что обновление должно производиться в режиме реального времени, для того, чтобы ответы модуля LLM были актуальными и релевантными.

4.5.1.4. Должны направляться оповещения об изменениях в базе знаний LLM для дальнейшей проверки обновления администратором базы знаний LLM и администратором диалоговых сценариев и промптов.

4.5.1.5. Должен фиксироваться процесс обновления базы знаний LLM (логирование).

4.5.1.6. Разметка СУЗ для работы с LLM должна проводиться разработчиком по согласованию с Заказчиком.

4.5.1.7. Для модуля Контакт-центра может быть создано неограниченное количество баз знаний LLM.

## 4.5.2. Сценарии и промпты

4.5.2.1. Разработчик должен создать и запустить в работу не менее 50 диалоговых сценариев до окончания процесса внедрения ПО. Примеры диалоговых сценариев в Приложении 2. Примеры сценариев могут быть актуализированы в дальнейшем в рамках технического задания.

4.5.2.2. Должен быть функционал для самостоятельного создания диалоговых сценариев администратором диалоговых сценариев и промптов, включая следующие возможности:

4.5.2.2.1. создание диалоговых сценариев любого уровня сложности (по количеству уровней и условий ветвления, интеграций с внешними системами, отправки, получения и обработки информации, получаемых из внешних систем, создания перечня уточняющих вопросов и т.п.);

4.5.2.2.2. редактирование действующих диалоговых сценариев, чтобы оперативно реагировать на изменения в банковских продуктах или потребности клиентов;

4.5.2.2.3. создание диалоговых сценариев на уровне визуального конструктора без использования языка программирования;

4.5.2.2.4. анализ эффективности текущих диалоговых сценариев и информирование о возможности оптимизации диалоговых сценариев с низкими показателями релевантности и полезности.

4.5.2.3. Должна поддерживаться работа диалоговых сценариев как в канале телефония, так и в онлайн-чате.

4.5.2.4. При завершении диалогового сценария должна фиксироваться обратная связь путём нажатия оператором канала телефония, оператором онлайн-чата одной из кнопок: «воспользовался», «не воспользовался», «неполный ответ» или иные.

4.5.2.5. В случае нечетких или нераспознанных ответов клиента в ходе сценариев должна предлагаться варианты уточняющих вопросов или передать разговор оператору онлайн-чата, если релевантность предполагаемого ответа низкая.

4.5.2.6. Должна быть возможность легко создавать, редактировать и удалять промпты без знаний языков программирования.

4.5.2.7. Должна быть возможность тестировать работу сценариев и промптов в отдельном функционале.

4.5.2.8. Должны формироваться результаты тестирования и в них должны отображаться ошибки в работе сценария или формулировке промпта.

## 4.5.3. Настройка релевантности

4.5.3.1. Разработчик должен реализовать интерфейс, позволяющий устанавливать уровни релевантности, которые определяют, насколько ответ модуля LLM соответствует ожиданиям пользователя.

4.5.3.2. Интерфейс по настройке уровней релевантности должен позволять:

4.5.3.2.1. устанавливать отдельные уровни для каждого канала связи (канал телефония, онлайн-чат) в зависимости от особенностей обслуживания;

4.5.3.2.2. тестировать уровни релевантности на основе исторических данных и анализа ответов, чтобы найти оптимальные значения для каждого типа канала и запроса;

4.5.3.2.3. регулировать уровни релевантности в зависимости от контекста или сложности запроса (например, требовать более высокий порог для финансовых или юридических запросов и более низкий для стандартных вопросов, таких как режим работы).

4.5.3.3. Интерфейс должен поддерживать функции автоматического отслеживания релевантности ответов, например, через отчеты о том, сколько запросов соответствует установленному уровню релевантности, и данные о случаях, когда модуль LLM не достиг минимального уровня.

## 4.6. Требования к интерфейсу модуля Контакт-центра

## 4.6.1. Интерфейс для внутренних пользователей Контакт-центра

4.6.1.1. Интерфейс для внутренних пользователей Контакт-центра должен представлять собой отдельное диалоговое окно с возможностью ведения диалога с модулем LLM через текстовые запросы, должны отображаться запросы и ответы модуля LLM.

4.6.1.2. Вместе с ответами модуля LLM должны отображаться ссылки на используемые статьи СУЗ и иные документы под каждым отдельным ответом.

4.6.1.3. Под каждым ответом модуля LLM должен отображаться процент релевантности ответа.

4.6.1.4. Интерфейс внутреннего пользователя Контакт-центра должен позволять выбирать на основе каких баз знаний LLM и/или групп баз знаний LLM проводится тестирование.

## 4.6.2. Интерфейс для операторов канала телефония

4.6.2.1. Интерфейс для операторов канала телефонии должен быть интегрирован непосредственно в действующее программное обеспечение телефонии либо представлен в виде отдельного окна разговора с клиентом.

4.6.2.2. В окне разговора с клиентом должны отображаться следующие атрибуты:

4.6.2.2.1. хронология преобразования разговора в текстовый формат;

4.6.2.2.2. тексты ответа модуля LLM на каждую реплику клиента (привязаны к соответствующим строчкам реплик клиента) с активными ссылками на используемые статьи СУЗ для просмотра оператором канала телефония и дополнительные материалы (изображения, таблицы, файлы и другие);

4.6.2.2.3. под каждым ответом модуля LLM должен отображаться процент релевантности ответа;

4.6.2.2.4. под каждым ответом модуля LLM должны располагаться кнопки «воспользовался», «не воспользовался», «неполный ответ» или иные для проставления оператором канала телефония отметки о полезности ответа;

4.6.2.3. Интерфейс должен позволять управлять окном с модулем суфлёра:

4.6.2.3.1. окно суфлёра должно позволять оператору канала телефония свернуть его или раскрыть, полностью отключить или вызвать снова после отключения;

4.6.2.3.2. окно суфлёра должно располагаться пропорционально окну разговора с клиентом и не мешать просмотру текста диалога и другой информации;

4.6.2.3.3. окно суфлёра должно отключаться при проставлении оператором канала телефония отметки, что предоставляется не консультация, а услуга, с возможностью повторного включения суфлёра в ходе разговора.

4.6.3.6.1. окно суфлёра должно позволять оператору онлайн-чата свернуть его или раскрыть, полностью отключить или вызвать снова после отключения;

4.6.3.6.2. окно суфлёра должно располагаться пропорционально окну разговора с клиентом и не мешать просмотру текста диалога и другой информации;

4.6.3.6.3. окно суфлёра должно отключаться при проставлении оператором онлайн-чата отметки, что предоставляется не консультация, а услуга, с возможностью повторного включения суфлёра оператором онлайн-чата в ходе разговора.

**4.6.4. Интерфейс базы знаний LLM**

4.6.4.1. Должен быть интуитивно понятный интерфейс и навигация.

4.6.4.2. Модуль Контакт-центра должен позволять редактировать базу знаний LLM, легко создавать промпты и диалоговые сценарии.

4.6.4.3. База знаний LLM должна отображать визуальную карту каждого диалогового сценария взаимодействия с клиентом, по которой можно определить последовательность и взаимосвязь вопросов и ответов.

4.6.4.4. Интерфейс базы знаний LLM должен включать диалоговое окно с модулем LLM для тестирования внутренним пользователем работы диалоговых сценариев, промптов и релевантности ответов модуля LLM на запросы.

4.6.4.5. Интерфейс базы знаний LLM должен включать отдельный раздел аналитики.

4.6.4.6. Интерфейс должен предоставлять возможность создания новых баз знаний LLM, их редактирования и удаления, в том числе компонентов баз знаний LLM.

**4.7. Требования к мониторингу и отчетности модуля Контакт-центра**

4.7.1. Модуль Контакт-центра должен иметь возможность для мониторинга всех этапов обработки запросов и корректности ответов.

4.7.2. Модуль Контакт-центра должен автоматически формировать отчеты с результатами по релевантности и полезности ответов модуля LLM на основе отметок, проставленных операторами канала телефония и онлайн-чата в разрезе ролей пользователей.

4.7.3. Модуль Контакт-центра должен включать систему мониторинга, которая позволит:

4.7.3.1. Отслеживать запросы в режиме реального времени:

4.7.3.1.1. при каждом обращении к модулю LLM фиксируются детали запроса, такие как канал обращения (канал телефония, онлайн-чат), содержание запроса, ключевые слова, предложенный ответ модуля LLM, отметку о полезности;

4.7.3.1.2. в случае отклонения ответа оператором канала телефония или оператора онлайн-чата или обнаружения низкой релевантности мониторинг фиксирует причину, чтобы дальнейшие ответы можно было оптимизировать.

4.7.3.2. Анализировать корректность ответов модуля LLM:

4.7.3.2.1. Модуль Контакт-центра должен отслеживать соответствие ответов модуля LLM ожиданиям и контексту запроса, включая показатели, такие как релевантность (в процентах) и отметки «воспользовался/не воспользовался/неполный ответ»;

4.7.3.2.2. данные по каждому ответу должны быть доступны для анализа с указанием причины низкой релевантности, если таковая была.

4.7.3.3. Контролировать производительность:

4.7.3.3.1. мониторинг должен фиксировать среднее и максимальное время отклика, отслеживать системные ресурсы, используемые для обработки запросов, и регистрировать случаи задержек в ответах, что позволит своевременно выявлять технические проблемы.

4.7.3.4. Оповещать об отклонениях:

4.7.3.4.1. в случае значительных отклонений от нормальных показателей (например, резкий рост запросов с низкой релевантностью или задержек в отклике) ПО должно автоматически отправлять уведомления специалистам для оперативного реагирования.

4.7.3.5. Выгрузка нераспознанных записей:

4.7.3.5.1. модуль отчетности должен предоставлять функционал для фильтрации и выгрузки нераспознанных записей, удовлетворяющих следующим критериям:

4.7.3.5.2. записи, где коэффициент успешности распознавания составляет менее 90%;

4.7.3.5.3. записи, содержащие 10% процентов неразборчивых фрагментов.

4.7.3.5.4. интерфейс выгрузки должен позволять выбирать диапазон дат, операторов канала телефония или онлайн-чата, номера телефонов, для фильтрации нераспознанных записей. Формат выгружаемых данных должен включать аудиофайл исходной записи, время начала и окончания каждого сегмента записи, транскрипт попытки распознавания, оценку качества распознавания в процентах.

4.7.4. Модуль Контакт-центра должен автоматически формировать отчеты и позволять настраивать отчеты самостоятельно (конструктор отчетов), которые включают:

4.7.4.1. Релевантность ответов:

4.7.4.1.1. отчет по среднему показателю релевантности для каждого канала (канал телефония, онлайн-чат);

4.7.4.1.2. разделение по типам запросов и категориям продуктов, позволяющее определить, в каких категориях модуль LLM достигает высокой релевантности, а где его необходимо доработать.

4.7.4.2. Оценка полезности ответов:

4.7.4.2.1. включение показателей полезности, отмеченных операторами канала телефония или онлайн-чата, таких как доля «воспользовался» и «не воспользовался», «неполный ответ» и иных ответов, чтобы выявить слабые места в работе модуля LLM;

4.7.4.2.2. сегментация этих данных по временным интервалам для анализа изменений в полезности ответов с течением времени.

4.7.4.3. Анализ ошибок и их причины:

4.7.4.3.1. Модуль Контакт-центра должен формировать отдельный отчет по ошибочным ответам, включая причины низкой релевантности (неполные данные, отсутствие нужного диалогового сценария), чтобы база знаний LLM и диалоговые сценарии могли быть обновлены.

4.7.4.4. Периодические отчеты для аналитиков и управленцев:

4.7.4.4.1. отчеты должны быть доступны в разных форматах (.xlsx, .pdf) и содержать сводные данные по каналам, качеству обслуживания и показателям LLM, чтобы их можно было использовать для улучшения клиентского сервиса.

4.7.4.5. Закономерности:

4.7.4.5.1. Модуль Контакт-центра должен анализировать тематику обращений за заданный промежуток времени и в режиме реального времени отображать на дашборде или отправлять уведомления о повторяющихся темах обращений за период.

4.7.4.6. Отчет по количеству повторных обращений:

4.7.4.6.1. Модуль Контакт-центра должен отображать данные о повторных обращениях с анализом их контекста (какое количество клиентов и тематика обращения).

4.7.4.7. Формирование отчетов по качеству преобразования речи в текст и анализу разговоров.

4.7.4.8. Отчеты по диалогам:

4.7.4.8.1. Модуль Контакт-центра должен формировать отчеты по каждому диалогу в виде отображения диалога в текстовом формате с репликами клиента и оператора канала телефония или онлайн-чата.

4.7.4.9. Модуль Контакт-центра должна формировать следующие настраиваемые виды отчетов:

4.7.4.9.1. категоризация запросов пользователей и клиентов, а также ответов модуля LLM по тематикам (анализ тематики как по запросу, так и по ответу);

4.7.4.9.2. отчет по диалогам с галлюцинациями LLM c возможностью просмотра запросов, полученных ответов и тематики;

4.7.5. Модуль Контакт-центра позволять строить статистические диаграммы, а также выгружать их в формате .png, .csv, .xlsx, .docх и других.

4.7.6. Модуль Контакт-центра должен генерировать отчеты регулярно (ежедневно, еженедельно, ежемесячно), а также по запросу.

4.7.7. Модуль Контакт-центра должен уметь формировать статистику в виде сводных таблиц, графиков, диаграмм.

4.7.8. Конструктор отчетов должен позволять:

4.7.8.1. настраивать содержание отчета;

4.7.8.2. добавлять или убирать определенные метрики;

4.7.8.3. изменять формат представления данных;

4.7.8.4. создавать собственные шаблоны отчетов для дальнейшего использования.

**5. Требования к модулю ИИ-ассистент**

**5.1. Общие требования к модулю ИИ-ассистент**

5.1.1. Модуль ИИ-ассистент должен включать функции модуля LLM.

5.1.2. Модуль ИИ-ассистент представляет собой интеллектуальный интерфейс взаимодействия пользователей с внутренними и внешними источниками данных, сервисами и инструментами на базе ИИ и должен распознавать, понимать и обрабатывать входящие запросы, чтобы предоставить пользователю релевантный ответ. Основные цели - упрощение доступа к информации, автоматизация рутинных задач, поддержка принятия решений, повышение производительности пользователей.

5.1.3. Модуль ИИ-ассистент должен обеспечивать:

5.1.3.1. наличие единого интеллектуального интерфейса: взаимодействие через текст (чат), поддержку как коротких запросов (промпт), так и сложных диалогов (контекстной беседы);

5.1.3.2. возможность персонализации интерфейса под пользователя или структурное подразделение банка.

5.1.4. Модуль ИИ-ассистент должен поддерживать возможность поиска и анализа информации как во внутренних источниках данных, так и во внешних источниках данных.

5.1.5. Модуль ИИ-ассистент должен поддерживать возможность интеграции с внутренними банковскими программными комплексами и системами.

5.1.6. Модуль ИИ-ассистент должен поддерживать сбор и агрегацию информации с автоматической сводкой.

5.1.7. Модуль ИИ-ассистент должен поддерживать возможность контекстной фильтрации и ранжирования по релевантности.

5.1.8. Модуль ИИ-ассистент должен поддерживать генерацию контента:

5.1.8.1. генерация текста - служебные/докладные записки, справки, инструкции, отчеты, ответы на обращения, описания процессов, доклады и иная текстовая бизнес-документация;

5.1.8.2. генерация презентаций -- с использованием утвержденных шаблонов, по заданной структуре (например -- цель/план/диаграммы/выводы);

5.1.8.3. генерация схем и диаграмм -- блок-схемы, архитектурные схемы, ER-диаграммы, BPMN-диаграммы;

5.1.8.4. генерация изображений.

5.1.9. Модуль ИИ-ассистент должен поддерживать создание документов в формате Word, PDF, Excel, PPT, BPMN.

5.1.10. Модуль ИИ-ассистент должен поддерживать интеграцию с RPA-сценариями (запуск процессов по команде).

5.1.11. Модуль ИИ-ассистент должен поддерживать функцию парсинга в соответствии с заданными пользователем параметрами.

5.1.12. Модуль ИИ-ассистент должен поддерживать функцию анализа документов и данных -- извлечение данных из документов, писем, файлов, возможность обработки вложений (PDF, Excel), построение резюме и аналитических выводов по загруженным материалам.

5.1.13. Модуль ИИ-ассистент должен обеспечивать возможность дообучения и адаптацию под терминологию и процессы банка, поддерживать обучение на истории взаимодействий.

5.1.14. Модуль ИИ-ассистент должен обеспечивать сохранность обрабатываемой информации и исключать возможность утечки персональных данных, банковской и коммерческой тайны.

5.1.15. Модуль ИИ-ассистент должен поддерживать настройку фильтров по типу контента (политика, религия и пр.).

5.1.16. Модуль ИИ-ассистент должен обеспечивать возможность получения статистики использования по подразделениям (в том числе в разрезе пользователей и ролей), типам запросов, оценки точности ответов, выгрузки логов и аналитических сводок.

5.1.17. Время обработки запроса с момента его поступления до предоставления ответа пользователю должно составлять не более 2 секунд.

5.1.18. Пользовательский интерфейс модуля ИИ-ассистент должен обеспечивать возможность направления запросов и получения ответов модуля LLM на основе базы знаний LLM, внутренних или внешних источников данных.

5.1.19. Модуль ИИ-ассистент должен фиксировать обратную связь путём нажатия пользователем одной из кнопок: «воспользовался», «не воспользовался», «неполный ответ» или иные.

5.1.20. Ответы модуля LLM должны содержать ссылки на использованные для формирования ответа источники.

5.1.21. Ответы модуля LLM должны генерироваться на том же языке, что и входящий запрос (русский или английский язык). В случае, если запрос пользователя направлен на одном из доступных языков и содержит требование перевода на другой доступный язык, ответ должен содержать:

5.1.21.1. ответ на английском языке: для непосредственного использования;

5.1.21.2. ответ на русском языке: для проверки корректности ответа и понимания пользователем информации.

5.1.22. Модуль ИИ-ассистент должен содержать функционал переводчика (перевод текста с/на английский язык).

5.1.23. В случае, если запрос пользователя не распознан, ПО должна выводить сообщение «запрос не распознан».

5.1.24. ПО должна предоставлять ответ модуля LLM пользователю на основе контекста предыдущих диалогов с данным пользователем с возможностью настройки автоматического использования или ручного вызова данного функционала.

5.1.25. Модуль ИИ-ассистент должен обеспечивать возможность отображения истории обращений пользователя, которая должна содержать сообщения за последние 10 дней.

5.1.26. Модуль ИИ-ассистент должен обеспечивать работу на основе требований к безопасности:

5.1.26.1. применяемая схема обеспечения конфиденциальности и целостности данных (передаваемых/хранящихся на серверной стороне) должна быть обеспечена с помощью SSL-сертификация (использование актуальных версий протоколов защиты);

5.1.26.2. модуль ИИ-ассистент должен разворачиваться в контуре Заказчика;

5.1.26.3. модуль ИИ-ассистент должен предоставлять возможность устанавливать ограничения на размер и тип передаваемых файлов.

5.1.27. Модуль ИИ-ассистент должен осуществлять отслеживание и фильтрацию операций, совершаемых пользователями в ПО (логирование).

5.1.28. Модуль ИИ-ассистент должен иметь возможность безопасного доступа к внешним источникам данных.

5.1.29. Модуль ИИ-ассистент должен поддерживать классификацию запросов по категориям (например -- операция, справка, инструкция).

5.1.30. Модуль ИИ-ассистент должен иметь возможности интеграции (при необходимости) с внутренними и внешними источниками данных.

5.1.31. Модуль ИИ-ассистент должен иметь возможности интеграции (при необходимости) с иными программными комплексами и системами банка.

5.1.32. Модуль ИИ-ассистент должен задавать уточняющие вопросы в случае поступления неясного запроса пользователя.

5.1.33. Доступ к модулю ИИ-ассистент должен осуществляться только после аутентификации пользователя.

5.1.34. Авторизация в модуле ИИ-ассистент должна быть реализована через AD.

5.1.35. Модуль ИИ-ассистент должен поддерживать одновременную работу не менее 2000 пользователей.

5.1.36. Для модуля ИИ-ассистент должен быть предусмотрен механизм для регулярного обучения на новых данных (документах, базах знаний LLM, внутренних и внешних источниках данных и пр.).

5.1.37. Для модуля ИИ-ассистент должна быть реализована панель администрирования для управления источниками данных, мониторинга работы ассистента и внесения изменений в его поведение.

5.1.38. Модуль ИИ-ассистент должен поддерживать функцию саммаризации информации из объемных документов, аудио- и видео записей.

5.1.39. Модуль ИИ-ассистент должен поддерживать функцию работы с программным кодом на основных языках программирования (генерация и проверка программного кода, SQL-запросов, тестирование, генерация тест-кейсов, перевод).

5.1.40. Модуль ИИ-ассистент должен поддерживать возможность замены LLM без изменения интерфейсов.

5.1.41. Модуль ИИ-ассистент должен предоставлять интерфейс управления базами знаний LLM. Интерфейс управления базами знаний LLM должен позволять пользователям без навыков использования языков программирования:

5.1.41.1. создавать произвольное число новых баз знаний LLM;

5.1.41.2. создавать произвольное количество записей в базах знаний LLM;

5.1.41.3. редактировать и удалять записи баз знаний LLM;

5.1.41.4. проводить обучение баз знаний LLM для управления модулем понимания запросов.

**5.2. Требования к модулю понимания запросов модуля ИИ-ассистент**

5.2.1. Модуль ИИ-ассистент должен включать в себя модуль понимания запросов пользователей. Модуль понимания запросов должен предоставлять возможность его настройки (управления) работниками банка без навыков использования языков программирования.

5.2.2. Для обучения модуль понимания запросов должен использовать обучающую выборку документов баз знаний LLM -- накопленные в каждом документе баз знаний LLM примеры вопросов пользователей, каждый новый вопрос пользователя должен автоматически сопоставляться с накопленным массивом примеров вопросов и ответов.

5.2.3. Для работы модуля понимания запросов (распознавания синонимичных слов и словосочетаний, полной смены формулировки на семантически близкую) должно быть достаточно одной формулировки вопроса.

5.2.4. Модуль понимания запросов должен поддерживать работу с русским и английским языками.

5.2.5. Модуль понимания должен автоматически учитывать порядок слов, использование синонимов или антонимов; результат каждого поиска в базе знаний LLM должен быть снабжен показателем релевантности в результатах поиска, отражающим вероятность того, что найденный документ базы знаний LLM соответствует вопросу пользователя.

5.2.6. Обучение модуля понимания не должно требовать указания ключевых слов или какой-либо другой ручной разметки примеров вопросов.

5.2.7. Обучение модуля понимания должно запускаться автоматически сразу после каждого обновления базы знаний LLM.

5.2.8. Добавление новых формулировок вопросов должно происходить полностью автоматически или через ручную валидацию администратором базы знаний LLM.

5.2.9. Список примеров в обучающей выборке должен также иметь возможность пополняться через загрузку из внутренних источников данных.

5.2.10. При добавлении новых формулировок модуль понимания должен автоматически контролировать наличие повторяющихся примеров в обучающей выборке и при пополнении обучающей выборки повторяющимся примером вопроса заблокировать его добавление.

5.2.11. Администратор базы знаний LLM должен иметь инструмент контроля над работой модуля понимания, который позволит определить какие документы будут найдены по запросу пользователя, с каким процентом релевантности и из какого примера вопроса.

5.2.12. Администратор базы знаний LLM должен иметь возможность указать минимальную релевантность поиска, начиная с которой модуль ИИ-ассистент может использовать найденный документ для предоставления ответа пользователю.

5.2.13. Должна быть возможность уточнения вопроса в случае, когда релевантности найденных документов недостаточно для прямого ответа, при этом должен предоставляться пользователю список вариантов ответа для выбора.

5.2.14. Должна быть возможность включить проверку и настройку действий в случае одинаковых реплик пользователя, поступивших в диалоге несколько раз подряд.

**5.3. База знаний модуля ИИ-ассистента:**

**разметка, работа с промптами**

**5.3.1. База знаний LLM**

5.3.1.1. Должна быть возможность добавлять и изменять промпты, сценарии, структуру и содержание базы знаний LLM без привлечения разработчика, без технических знаний, навыков, без знания языков программирования.

5.3.1.2. Должна быть возможность редактировать и структурировать данные базы знаний LLM так, чтобы их эффективно могла использовать LLM-модель.

5.3.1.3. Должна обновляться база знаний LLM в соответствии с изменениями во внутренних источниках данных любыми способами, например, API, учитывая, что обновление должно производиться в режиме реального времени, для того, чтобы ответы модуля LLM были актуальными и релевантными.

5.3.1.4. Должны направляться оповещения об изменениях в базе знаний LLM для дальнейшей проверки обновления администратором базы знаний LLM и администратором диалоговых сценариев и промптов.

5.3.1.5. Должен фиксироваться процесс обновления базы знаний LLM (логирование).

5.3.1.6. Разметка внутренних источниках данных для работы с LLM должна проводиться разработчиком по согласованию с банком.

5.3.1.7. Для модуля ИИ-ассистент может быть создано неограниченное количество баз знаний LLM.

**5.3.2. Промпты**

5.3.2.1. Должен быть функционал для самостоятельного создания промптов.

5.3.2.2. Функционал должен позволять легко создавать, редактировать и удалять промпты без знаний языков программирования.

**5.3.3. Настройка релевантности**

5.3.3.1. Разработчик должен реализовать интерфейс, позволяющий устанавливать уровни релевантности, которые определяют, насколько ответ модуля LLM соответствует ожиданиям пользователя.

5.3.3.2. Интерфейс должен поддерживать функции автоматического отслеживания релевантности ответов, например, через отчеты о том, сколько запросов соответствует установленному уровню релевантности, и данные о случаях, когда модуль LLM не достиг минимального уровня.

**5.4. Требования к отчетности**

5.4.1. Модуль ИИ-ассистент должен иметь возможность автоматического формирования отчетов с результатами по релевантности и полезности ответов модуля LLM на основе отметок, проставленных пользователями ИИ-ассистента, в том числе:

5.4.1.1. Релевантность ответов:

5.4.1.1.1. отчет по среднему показателю релевантности;

5.4.1.1.2. разделение по типам запросов, позволяющее определить, в каких категориях модуль LLM достигает высокой релевантности, а где его необходимо доработать.

5.4.1.2. Оценка полезности ответов:

5.4.1.2.1. включение показателей полезности, отмеченных пользователями ИИ-ассистента, таких как доля «воспользовался» и «не воспользовался», «неполный ответ» и иных ответов, чтобы выявить слабые места в работе модуля LLM;

5.4.1.2.2. сегментация этих данных по временным интервалам для анализа изменений в полезности ответов с течением времени.

5.4.1.3. Анализ ошибок и их причины:

5.4.1.3.1. Модуль ИИ-ассистент должен формировать отдельный отчет по ошибочным ответам, включая причины низкой релевантности (неполные данные, отсутствие нужного диалогового сценария).

5.4.2. Модуль ИИ-ассистент должен формировать следующие настраиваемые виды отчетов:

5.4.2.1. категоризация запросов пользователей, а также ответов модуля LLM по тематикам (анализ тематики как по запросу, так и по ответу);

5.4.2.2. отчет по диалогам с галлюцинациями LLM c возможностью просмотра запросов, полученных ответов и тематики.

5.4.3. Модуль ИИ-ассистент должен генерировать отчеты регулярно (ежедневно, еженедельно, ежемесячно), а также по запросу.

5.4.4. Модуль ИИ-ассистент должен уметь формировать статистику в виде сводных таблиц, графиков, диаграмм.

5.4.5. Конструктор отчетов должен позволять:

5.4.5.1. настраивать содержание отчета;

5.4.5.2. добавлять или убирать определенные метрики;

5.4.5.3. изменять формат представления данных;

5.4.5.4. создавать собственные шаблоны отчетов для дальнейшего использования.

**5.5. Обучение модулю ИИ-ассистента**

5.5.1. Разработчик должен предоставить полный пакет обучения для пользователей по функционалу модуля ИИ-ассистент: обучающие материалы, технические и пошаговые инструкции, руководства пользователя по каждой роли в отдельности, консультации.

5.5.2. Разработчик должен охватить в обучении следующие направления:

5.5.2.1. работа с интерфейсом модуля ИИ-ассистент;

5.5.2.2. обзор функционала по каждой роли в отдельности;

5.5.2.3. создание и изменение промптов;

5.5.2.4. настройка релевантности;

5.5.2.5. работа с базой знаний LLM;

5.5.2.6. работа с отчетностью;

5.5.2.7. работа с программным обеспечением и администрирование ПО.

**6. Требования к модулю распознавания документов.**

**6.1. Общие требования к модулю распознавания документов.**

6.1.1. Модуль распознавания документов должен включать функции модуля LLM.

6.1.2. Модуль распознавания документов предназначен для автоматического распознавания, классификации и извлечения структурированной информации из различных типов документов, поступающих в банк в виде фотографий, PDF-файлов. Модуль должен использовать технологии компьютерного зрения и искусственного интеллекта для повышения точности и скорости обработки документов.

6.1.3. Модуль распознавания документов должен поддерживать работу со следующими видами документов:

6.1.3.1. документы, удостоверяющие личность;

6.1.3.2. анкеты, заявления;

6.1.3.3. справки;

6.1.3.4. договоры (включая кредитные, депозитные, аренды и т.д.);

6.1.3.5. платежные документы (квитанции, чеки, счета);

6.1.3.6. уставные документы юр. лиц (устав, выписки, лицензии и пр.);

6.1.3.7. банковские выписки;

6.1.3.8. документы на жилье, имущество;

6.1.3.9. документы, оформленные вручную (от руки).

6.1.4. Модуль распознавания документов должен обеспечивать возможность загрузки документов через API, web-интерфейс и интеграцию с ПО электронного документооборота.

6.1.5. Модуль распознавания документов должен поддерживать пакетную загрузку.

6.1.6. Модуль распознавания документов должен поддерживать различные форматы документов: .pdf, .jpg, .png, .tiff, .jpeg и иные.

6.1.7. Модуль распознавания документов должен поддерживать автоматическое определение типа документа с возможностью ручной переквалификации типа документа пользователем.

6.1.8. Модуль распознавания документов должен поддерживать оптическое распознавание текста (OCR).

6.1.9. Модуль распознавания документов должен поддерживать высокоточное извлечение текста из изображений и PDF-документов.

6.1.10. Модуль распознавания документов должен поддерживать распознавание как печатного, так и рукописного текста.

6.1.11. Модуль распознавания документов должен обеспечивать поддержку многостраничных документов.

6.1.12. Модуль распознавания документов должен поддерживать работу с многоязычными документами (русский, английский).

6.1.13. Модуль распознавания документов должен содержать функционал извлечения ключевых полей, выделения и структурирования информации из документа (ФИО, даты, серии и номера документов, суммы, реквизиты, наименования компаний, адреса и пр.), а также возможность настройки пользовательских шаблонов под документы.

6.1.14. Модуль распознавания документов должен обеспечивать проверку качества и валидацию данных, проверку полноты распознавания (например, заполненность всех обязательных полей), проверку корректности извлечённых данных (по справочникам, регулярным выражениям и т.д.), выявление аномалий, ошибок или несоответствий.

6.1.15. Модуль распознавания документов должен обеспечивать наличие визуального интерфейса для просмотра распознанного текста поверх оригинала, возможность ручной корректировки данных пользователем, подсветку сомнительных зон.

6.1.16 Модуль распознавания документов должен позволять экспортировать распознанные документы в форматы: pdf, docx, .txt, .xlsx.

6.1.17. Модуль распознавания документов должен иметь возможность интеграции с внутренними системами банка.

6.1.18. Точность OCR модуля распознавания документов должна составлять не менее 95% на стандартных отсканированных документах.

6.1.19. Производительность модуля распознавания документов должна составлять не менее 1 страницы в 1 секунду на 1 поток.

6.1.20. Модуль распознавания документов должен иметь возможность обучения на новых типах документов.

6.1.21. Модуль распознавания документов должен обеспечивать шифрование передаваемых и хранимых данных, аудит действий пользователя, поддержку разграничений прав доступа.

6.1.22. Модуль распознавания документов должен иметь возможность локального развертывания и поддерживать работу в изолированной сети.

6.1.23. Модуль распознавания документов должен обеспечивать ведение логирования всех операций распознавания, формирования отчетов о статусе обработки (успех, ошибки, отказ) и возможность выгрузки данных в Excel, CSV, JSON.

6.1.24. Модуль распознавания документов должен иметь возможность интеграции чат-ассистентами (для подтверждения распознавания данных).

6.1.25. Модуль распознавания документов должен иметь возможность включения в цепочку обработки RPA (как часть процесса).

**6.2. Требования к отчетности**

6.2.1. Модуль распознавания документов должен иметь возможность автоматического формирования отчетов с результатами распознавания документов, в том числе в части результатов распознавания (успешно, с ошибками, отказ) и статистики использования (типы документов, пользователи).

6.2.2. Модуль распознавания документов должен формировать отдельный отчет по распознаваниям с ошибками, включая причины распознавания с ошибками.

6.2.3. Модуль распознавания документов должен генерировать отчеты регулярно (ежедневно, еженедельно, ежемесячно), а также по запросу.

6.2.4. Модуль распознавания документов должен уметь формировать статистику в виде сводных таблиц, графиков, диаграмм.

6.2.5. Конструктор отчетов должен позволять:

6.2.5.1. настраивать содержание отчета;

6.2.5.2. добавлять или убирать определенные метрики;

6.2.5.3. изменять формат представления данных;

6.2.5.4. создавать собственные шаблоны отчетов для дальнейшего использования.

**10. Системные требования.**

10.2. На серверах допускается использование виртуализации. ПО должно использовать виртуализацию VMware (за исключением высоконагруженных серверов).

10.3. ПО должно использовать серверные операционные системы семейства Linux, не имеющими статуса End of Life по информации на сайте производителя.

10.4. При необходимости использования СУБД:

10.4.1. СУБД должна быть свободно распространяемой (СУБД с открытым исходным кодом, open source решение);

10.4.2. ПО должно использовать СУБД с использованием сетевого протокола TCP/IP и обеспечивающих поддержку консистентности информации при одновременной работе большого числа пользователей.

**11. Требования к документации.**

При этом моментом получения или решения обращения будет считаться момент получения запрашиваемой дополнительной (уточняющей) информации.

Запросы по получению дополнительной (уточняющей) информации регистрируются непосредственно в обращениях в виде комментариев к обращению или направляются в письменной форме посредством электронной почты, а электронные копии прикрепляются к обращению.

В случае получения от Исполнителя запроса в письменной форме, Заказчик обязан предоставить Исполнителю ответы на поставленные вопросы в письменной форме посредством электронной почты и прикрепить электронную копию ответа к обращению.

2.5.
