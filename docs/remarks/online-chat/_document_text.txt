1. Контекст и цели

1.1. Контекст

Контакт-центр банка обслуживает клиентов по телефонии (ПТК/Oktell) и онлайн-каналам (чат на сайте, мессенджеры, социальные сети).

По договору Исполнитель поставляет собственную операционную платформу онлайн-чата в контуре банка: прием обращений, маршрутизация, рабочее место оператора, администрирование, отчётность.

Платформа чата работает совместно с модулем суфлера: оператор получает подсказки на основе СУЗ; клиенту в чате ссылки на статьи СУЗ не отправляются (требование проекта).

Для операторов КЦ доступ в интернет ограничен; исключение — функционал, необходимый для работы онлайн-чата (каналы связи с клиентами), в соответствии с Приложением 1.

2. Глоссарий и роли

2.1. Основные термины

Термин

Определение

Клиент / посетитель

Физическое или юридическое лицо, обращающееся в банк через онлайн-канал

Обращение / диалог

Переписка клиента с банком в рамках одной темы; может включать несколько сессий

Сессия обращения

Логическая связка обращений одного клиента; хранит контекст для истории и суфлера

Канал

Способ связи: виджет на сайте, Telegram, Viber, ВКонтакте, Одноклассники, API-канал

Размещение

Набор настроек виджета (внешний вид, тексты, отдел, код встраивания) для конкретных страниц сайта

Отдел

Группа операторов для обработки определённого типа запросов (продукт, тема, язык)

Очередь

Список диалогов, ожидающих назначения или ответа оператора

Автоназначение

Автоматическое распределение новых диалогов на свободных операторов по правилам

Рабочее место оператора (АРМ)

Основной интерфейс оператора: список диалогов, переписка, данные клиента

Панель управления

Административный интерфейс: операторы, отделы, размещения, настройки

Суфлёр

Модуль подсказок оператору на основе СУЗ (RAG + LLM); встраивается в АРМ чата

Офлайн-обращение

Диалог, в котором клиент не в сети; обрабатывается по отдельным правилам очереди

Бот

Автоматизированный обработчик первой линии (приветствие, маршрутизация, FAQ)

5.1. Сценарии клиента

UC-K1. Клиент начинает чат на сайте

Участники

Клиент, виджет, ядро платформы, оператор

Предусловия

Виджет размещён на странице сайта; есть операторы в статусе «онлайн»

Слайды

1–3

Основной поток: 1. Клиент нажимает кнопку чата (FAB) на странице сайта. 2. Открывается окно виджета; при необходимости клиент выбирает тему и заполняет контактные поля. 3. Клиент нажимает «Начать чат» и отправляет первое сообщение. 4. Система создаёт обращение, направляет в отдел по правилам маршрутизации, назначает оператора или ставит в очередь. 5. Клиент видит системное сообщение о подключении оператора и примерном времени ответа. 6. Оператор отвечает; клиент получает сообщение в виджете.

Альтернативы: нет свободных операторов — клиент видит статус очереди-ранее писала что с возможностью настройки; клиент закрывает окно до ответа — см. UC-K4. (офлайн диалог становится только при ожидании заданного времени,   если клиент не дождался ответа – то это либо потерянный либо отказ от консультирования. Офлайны  - как правило идут только в нерабочее время. В рабочее время – это потерянные или отказы ( клиент передумал консультироваться – таймер по отказу до 60 секунд может быть).

Результат: обращение зарегистрировано, диалог ведется в виджете.

UC-K2. Клиент пишет в мессенджер

Участники

Клиент, адаптер канала, ядро, оператор

Предусловия

Канал (Telegram / Viber / ВКонтакте / Одноклассники) подключен администратором

Слайды

4

Основной поток: 1. Клиент отправляет сообщение в мессенджер банка. 2. Адаптер канала передает сообщение в ядро платформы. 3. Обращение появляется в единой очереди АРМ с указанием канала. 4. Оператор открывает диалог и отвечает из АРМ. 5. Ответ доставляется клиенту в тот же мессенджер.

Альтернативы: ограничения API канала (цитирование, вложения) — оператор видит доступные действия.

Результат: клиент получил ответ в привычном мессенджере.

UC-K3. Клиент повторно обращается

Участники

Клиент, оператор, единая история

Предусловия

Ранее был диалог с этим клиентом; идентификатор клиента определён

Слайды

3–4

Основной поток: 1. Клиент пишет повторно (тот же канал или другой согласованный канал). 2. Система сопоставляет клиента по согласованному идентификатору (телефон, ID и т.д.). 3. Создаётся новый номер диалога с привязкой к истории клиента. 4. Оператор видит прошлую переписку, комментарии коллег и контекст точки входа.

Результат: непрерывность обслуживания без потери контекста.UC-K4. Клиент офлайн после начала диалога

Участники

Клиент, оператор

Предусловия

Диалог начат, клиент покинул сайт или мессенджер

Слайды

4

Основной поток: 1. Система фиксирует отсутствие клиента в сети. 2. Диалог переходит в раздел «Оффлайн-обращения» в АРМ. 3. Оператор формирует и отправляет ответ. 4. При возврате клиента сообщение отображается в виджете/мессенджере; при свернутом виджете — бейдж на FAB.

Результат: ответ клиента не потерян.

5.2. Сценарии оператора

UC-O1. Оператор принимает назначенный диалог

Участники

Оператор, ядро

Предусловия

Оператор авторизован через AD; статус «онлайн»

Слайды

4

Основной поток: 1. Оператор входит в АРМ и устанавливает статус «онлайн». 2. Новое обращение появляется в разделе «Ожидают ответа» (автоназначение или очередь). 3. Оператор открывает карточку диалога; в центре — лента сообщений, справа — данные клиента. 4. Таймер SLA сбрасывается при начале работы оператора.

Результат: оператор ведет назначенный диалог.

UC-O2. Оператор отвечает с использованием суфлера

Участники

Оператор, суфлёр, клиент, СУЗ

Предусловия

Активный диалог; индекс СУЗ актуален

Слайды

4

Основной поток: 1. Клиент отправляет вопрос. 2. Ядро передает событие диалога в модуль суфлера. 3. В правой панели АРМ появляются ранжированные подсказки с % релевантности. 4. Оператор нажимает «Вставить» или редактирует текст вручную. 5. Оператор отправляет ответ клиенту без URL статей СУЗ. 6. При необходимости оператор открывает статью СУЗ только у себя (ссылка не уходит клиенту).

Альтернативы: суфлёр недоступен — оператор отвечает без подсказок; подсказка неполная — обратная связь «Неполно».

Результат: клиент получил корректный ответ; фидбек суфлера сохранён.

UC-O3. Оператор переводит диалог

Участники

Оператор, коллега / супервизор, ядро

Слайды

4

Основной поток: 1. Оператор нажимает «Перевести» в зоне переписки. 2. Выбирает другого оператора, отдел или очередь. 3. Диалог исчезает из списка текущего оператора. 4. У получателя диалог появляется в очереди с сохранением истории и контекста.

Результат: обращение передано без потери данных.

UC-O4. Оператор закрывает диалог

Участники

Оператор

Слайды

4

8. Интеграция с суфлером и смежными системами

8.1. Суфлёр ↔ платформа чата

Направление

Содержание

Чат → Суфлёр

Каждое сообщение клиента в активном диалоге передается для формирования подсказки (идентификатор сессии, канал, текст, время)

Суфлёр → АРМ

Ранжированные подсказки, процент релевантности, ссылки на статьи СУЗ для оператора

Оператор → Суфлёр

Обратная связь: подсказка использована / неполная / отключить суфлер для диалога

Ограничение

Текст, отправляемый клиенту, не содержит URL статей СУЗ

8.2. Единая история клиента

Все диалоги платформы чата записываются в единую историю взаимодействий проекта (срок хранения — по договору, ориентир 1 год).

Суфлёр использует историю для контекста подсказок.

Идентификатор клиента согласуется с моделью банка (телефон, ID клиента, внешний ключ CRM).
