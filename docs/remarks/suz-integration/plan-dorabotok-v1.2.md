# План доработок к единому ТЗ v1.2 (СУЗ ↔ суфлёр)

Основание: отработанный файл `TZ-Bitrix-RAG +-ответы-исполнителя.docx` и сводка `otvety-ponomarev.md`.

## 1) Цель и границы

- Подготовить раздел интеграции для включения в единое ТЗ v1.2.
- Закрыть замечания заказчика по терминологии, модели обмена, приёмке и ответственности.
- В границах этого плана: только содержание ТЗ и критерии приёмки интеграции.
- Вне границ: разработка/внедрение кода Bitrix и суфлёра, а также организационные вопросы ДИТ.

## 2) Ключевой результат

- Финальный раздел `Интеграция СУЗ ↔ суфлёр` в едином ТЗ v1.2 с:
- единым глоссарием и ссылками на ТТ;
- описанием моделей интеграции A/B и обоснованием выбора B;
- разделением интеграционной приёмки и приёмки функционала КЦ;
- согласованной матрицей BTX (что штатно, что доработка).

## 3) Перечень доработок

### 3.1 Термины, роли, ссылки на ТТ

- Привести названия и определения к Прил. 1 ТТ (СУЗ, Суфлёр, роли §2.4).
- Убрать разнобой терминов (`CMS`, `scope`, «подразделение ЦК» и т.п.).
- Зафиксировать правило: термины из ТТ дублируются в глоссарии; новые — описываются дополнительно.
- Добавить трассировку: таблица `пункт Прил. 1 → раздел ТЗ → сценарий → приёмка`.

### 3.2 Модель интеграции

- Восстановить и зафиксировать в ТЗ обе модели:
- **Модель A**: короткий webhook + GET статьи;
- **Модель B**: webhook с полным телом.
- Зафиксировать выбор **модели B** как рабочей (GET только INT-10 и опц. INT-06/fallback).
- Убрать/переформулировать устаревшие фрагменты, где GET описан как обязательный после каждого события.

### 3.3 Состав данных и бизнес-правила

- Явно описать payload (§6.2): `event_id`, `status`, `version_id`, `permalink`, `visibility_scope`, `body_html/body_plain`.
- Уточнить источники для RAG: статьи СУЗ, HTML-разметка в теле статьи; вложения в v1 не индексируются.
- Закрепить, что `visibility_scope` вычисляется модулем sync по правилам банка (нет штатной галки «только КЦ»).
- Развести статусы `draft/published/archived/unpublished/deleted` и их влияние на индекс.

### 3.4 SLA и приёмка

- Зафиксировать: `§4.6` — только приёмка интеграционного обмена (webhook/outbox/payload/idempotency).
- Вынести UI/качество подсказок/операторские сценарии в отдельную приёмку раздела «Суфлёр».
- Уточнить формулировки по времени:
- по ТТ: «реальное время/онлайн» для обновления базы знаний;
- `< 2 мин` — только верхняя граница интеграционного теста доставки webhook.

### 3.5 ИБ и ответственность сторон

- Использовать терминологию ТТ: «требования информационной безопасности (ИБ)».
- Зафиксировать разделение ответственности:
- Заказчик: инфраструктура/канал/согласование ИБ;
- Bitrix: события/outbox/payload по BTX;
- Исполнитель: индексация, RAG, LLM, оповещения, методика разметки.

## 5) Порядок выполнения

1. Синхронизировать глоссарий и роли с ТТ.
2. Обновить разделы с моделью интеграции (A/B + выбор B).
3. Уточнить payload/INT/BTX и правила статусов/области индексации.
4. Перестроить §4.6 (только интеграционный контур).
5. Пройти контрольный чек-лист по ссылкам на ТТ и терминологии.
6. Подготовить версию на согласование.

## 6) Критерии готовности

- Нет конфликтов терминов с ТТ.
- Для каждого ключевого утверждения в интеграционном разделе есть ссылка на пункт ТТ.
- Модель B зафиксирована как рабочая; модель A сохранена как сравнительная/резервная.
- §4.6 не смешивает интеграционную и функциональную приёмку КЦ.
- Все замечания из `TZ-Bitrix-RAG +-ответы-исполнителя.docx`, относящиеся к интеграционному разделу, отражены в тексте v1.2.
