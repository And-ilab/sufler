## Цель v1.4

Закрыть замечания заказчика после передачи v1.3:

1. **Макеты ↔ ТТ** — интерфейсы соответствуют листовым пунктам §4.3–4.7.
2. **Трассировка** — каждый листовой пункт ТТ имеет FR/UC + строку в Прил. D.
3. **Сквозная логика** — каналы, статусы, API, разделение суфлёр/ассистент/Hub.
4. **Приёмка** — проверяемые критерии по каждому FR/UC (формат I.1.2).
5. **Тестовые сценарии** — каждый ID приёмки (SUF-T, CHAT-T, ASS-T, INT-T) и UC/FR имеет **развёрнутое описание** шагов проверки, а не только ссылку на идентификатор.

Не перенос формулировок ТТ — описание реализации: вход → под капотом → выход → приёмка.

## Этап 1.5 — тестовые сценарии приёмки (P0)

**Проблема v1.3:** во многих FR/UC указаны только идентификаторы приёмки (`SUF-T-01`, `CHAT-T-04`, `ASS-T-05`, `INT-T-*`) без пошагового сценария проверки. Протокол 06.07 **п. 9**: пересмотреть и унифицировать критерии приёмки по всем разделам.

### Объём аудита

| Область | Разделы ТЗ v1.3 | ID-префиксы |
|---------|-----------------|-------------|
| Контакт-центр (суфлёр + чат) | II.7 | SUF-T-01…14, CHAT-T-01…20 |
| Ассистент | III.8 | ASS-T-* |
| Документы | IV.8 | DOC-T-* |
| Интеграции | VI | INT-T, INT-T-AUD |
| Диалоговые сценарии КЦ | Прил. B, [manifest.yaml](../sources/technical-requirements/app2-scenarios/manifest.yaml) | CC-SCR-001…010 (+ backlog ≥40) |

### Обязательная структура каждого тестового сценария

Каждый *-T-* и каждый UC с блоком «Приёмка» должен содержать **не только ID**, а развёрнутое описание:

| Поле | Содержание |
|------|------------|
| **ID** | SUF-T-01, CHAT-T-04 и т.д. |
| **Название** | Краткая формулировка проверки |
| **Предусловия** | Роль, стенд, данные, конфигурация |
| **Шаги** | Пронумерованные действия тестировщика |
| **Ожидаемый результат** | Наблюдаемый и измеримый итог |
| **Стенд** | test Oktell / test Bitrix / Hub staging / Postman и т.д. |
| **Связь** | FR/UC + **листовой** пункт ТТ (§4.x.x.x) |
